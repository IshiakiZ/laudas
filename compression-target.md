# Laudas compression target — 5-10× via spec-first authoring

> **Goal:** typical scripts written in Laudas (spec-first) are **5-10× shorter** than equivalent Python (production code + tests combined). Hand-written body code (no spec-first) is **2-4× shorter**.

This document explains how, with a concrete worked example, and what language features have to ship for the 10× story to hold consistently.

---

## 1. The honest comparison

The 10× figure is real, but only if the comparison is fair:

| What you write | Python | Laudas |
|---|---|---|
| Production code | yes | minimal (spec only in spec-first mode) |
| Unit tests | separate file | folded into `ex` slots |
| Type validation | manual or `pydantic` | refinement types in `in` / `out` |
| Behavior docs | docstring | `prose` slot — LLM-checked, doubles as spec |

In a real Python project the *test code* is often equal in size to the *production code*. Laudas collapses these into one artifact: the spec. That's the **2× baseline** before anything else.

---

## 2. Worked example — CSV → validate → dedupe → sort → JSON

Task: read a CSV of users (`name`, `email`, `role`), drop rows missing fields or malformed email, dedupe by lowercased email, sort by (role, name), write JSON.

### 2.1 Python (~38 lines production + ~30 lines tests = ~68 lines)

```python
import csv, json, sys
from dataclasses import dataclass

@dataclass
class UserRecord:
    name: str
    email: str
    role: str

def parse_csv(path):
    records = []
    with open(path, encoding='utf-8') as f:
        for row in csv.DictReader(f):
            name = row.get('name', '').strip()
            email = row.get('email', '').strip()
            role = row.get('role', '').strip()
            if not name or not email or not role:
                continue
            if '@' not in email:
                continue
            records.append(UserRecord(name, email, role))
    return records

def dedupe_by_email(records):
    seen, out = set(), []
    for r in records:
        key = r.email.lower()
        if key not in seen:
            seen.add(key)
            out.append(r)
    return out

def main(in_path, out_path):
    records = parse_csv(in_path)
    records = dedupe_by_email(records)
    records.sort(key=lambda r: (r.role, r.name))
    with open(out_path, 'w', encoding='utf-8') as f:
        json.dump([r.__dict__ for r in records], f, indent=2)

if __name__ == '__main__':
    main(sys.argv[1], sys.argv[2])

# Plus ~30 lines of pytest unit tests covering missing fields,
# malformed email, dedupe behavior, sort order, file I/O.
```

### 2.2 Laudas spec-first (10 lines total)

```laudas
type UserRecord { name: str, email: str, role: str }

fn process_users
vis appearing
eff io,fails
in in_path: str
in out_path: str
out unit
prose "Read CSV at in_path with name/email/role columns. Skip rows missing fields or whose email lacks '@'. Dedupe by lowercased email. Sort by (role, name). Write JSON to out_path with 2-space indent."
do
end
```

**10 lines · 6.8× compression** vs the combined Python production + tests. The body is generated by `mira request-body` and verified by `voronin`. The user-authored artifact is the spec — and the spec is also the test.

### 2.3 Laudas hand-written body (12 lines)

```laudas
type UserRecord { name: str, email: str, role: str }

fn process_users
vis appearing
eff io,fails
in in_path: str
in out_path: str
out unit
do
let users = text.parse_csv(archive.read(in_path))
  .filter(r -> r.name && r.email && "@" in r.email && r.role)
  .map(r -> UserRecord { name: r.name, email: r.email, role: r.role })
  .dedupe_by(u -> u.email.lower())
  .sort_by(u -> (u.role, u.name))
archive.write(out_path, text.to_json(users))
end
```

**12 lines · 5.7× compression** even without spec-first inversion.

## 3. Where the compression comes from

1. **Spec is the test.** `ex`, `req`, `ens` slots collapse Python production code + pytest tests into one artifact. ~**2× baseline.**
2. **Strong stdlib of verified primitives.** `text.parse_csv`, `ledger.dedupe_by`, `ledger.sort_by`, `text.to_json` each replace 5-15 lines of Python. Composes with #5.
3. **Method chaining.** No intermediate variable names. `data.filter(p).map(f).sort_by(g)` is the whole pipeline.
4. **Arrow lambdas.** `r -> r.email.lower()` is shorter than Python's `lambda r: r.email.lower()` and reads naturally inside chains.
5. **Spec-first inversion.** When `do` is empty, the body is derivable. The user-authored artifact is just the spec.
6. **Type aliases for records.** `type UserRecord { name: str, … }` replaces `@dataclass class UserRecord:`.
7. **Refinement types in signatures.** Validation lives in the type, not as `assert` statements at the top of each function body.
8. **Effects in signatures.** No need for separate `Result` / `Either` / try-except boilerplate.

## 4. What v0 has — and what the 10× story still needs

| Feature | v0 | v0.1 | v0.5 | Compression critical? |
|---|---|---|---|---|
| Wire-format parser | ✓ | ✓ | ✓ | yes (foundation) |
| Example runner | ✓ | ✓ | ✓ | yes (spec = test) |
| Stratified spec (`ex`/`prose`/`req`/`ens`) | ✓ | ✓ | ✓ | yes (spec = test) |
| Z3 verification | — | ✓ | ✓ | indirectly (enables spec-first trust) |
| Spec-first inversion (`mira request-body`) | — | — | ✓ | **yes (10× story)** |
| Strong stdlib (`text`, `ledger`, `archive`, …) | — | — | ✓ | **yes (10× story)** |
| Method chaining | — | — | ✓ | yes |
| Arrow lambdas (`r -> ...`) | — | — | ✓ | yes |
| Type aliases for records | — | — | ✓ | yes |
| Pattern destructuring | — | — | v1.0 | nice-to-have |

The 10× story is a **v0.5 milestone**. v0 today demonstrates the spec-as-test half (~2× already, on toy programs). v0.1 adds Z3 to make spec-first trustworthy. v0.5 adds the syntactic and library features that close the gap.

---

## 5. When 10× does *not* apply

- **Heavy algorithmic code with no tests** — Python wasn't paying the test tax, so the comparison shrinks. Expect 2-3×.
- **Glue code that's already short in Python** — diminishing returns; expect 1.5-2×.
- **Anything blocked on a missing stdlib primitive** — until the relevant `ledger` / `transit` / `text` function exists, that part doesn't compress.

The honest summary: **5-10× for typical real-world scripts with proper specs. 2-4× for raw algorithmic code that wasn't paying the test tax in Python.**
